## 概要

このリリースの目玉は、`aws-mediaconnect` の L2 コンストラクト追加と、合成した CloudFormation テンプレートを WASM ベースの Linter (`CloudFormationValidatePlugin`) で自動検証する新しい仕組みです。加えて CloudFront の Managed-HostHeaderOnly オリジンリクエストポリシー、ECS の Amazon Linux 2023 (Neuron) AMI サポート、`bedrockagentcore-alpha` の Gateway IAM 認証プロバイダー拡張などの新機能が追加されました。バグ修正ではトークン化された値をバリデーションが誤って拒否していた問題 (Lambda / Backup) 、`stack.node.addDependency` のスタック間依存関係における性能改善などが含まれます。

**リリース:** [v2.262.0](https://github.com/aws/aws-cdk/releases/tag/v2.262.0)

なお、`feat(apigateway): alb integration` ([#36247](https://github.com/aws/aws-cdk/pull/36247)) と `fix(lambda): allow SnapStart for container image functions` ([#38265](https://github.com/aws/aws-cdk/pull/38265)) は同じリリース内で revert されています (詳細は「Revert / 未提供機能」を参照) 。

## 新機能

### `mediaconnect`: L2 コンストラクトを新規追加 ([#37945](https://github.com/aws/aws-cdk/pull/37945))

**この機能でできること:**
- AWS Elemental MediaConnect の Flow / Source / Output などを L2 コンストラクトで扱えるようになりました。これまで `Cfn*` (L1) しか提供されていなかった MediaConnect が、型安全な API とデフォルト値を持つ L2 として利用可能になっています。
- ライブ動画伝送パイプラインを CDK でモジュラーに定義でき、Grant API / IAM の連携も容易になります。

**使用例:**

```typescript
import * as mediaconnect from 'aws-cdk-lib/aws-mediaconnect';
import { Stack } from 'aws-cdk-lib';
import { Construct } from 'constructs';

export class MediaConnectStack extends Stack {
  constructor(scope: Construct, id: string) {
    super(scope, id);

    // MediaConnect Flow を作成
    const flow = new mediaconnect.Flow(this, 'Flow', {
      // ソース設定 (RTP-FEC などのプロトコルを指定)
      source: {
        name: 'PrimarySource',
        protocol: mediaconnect.Protocol.RTP_FEC,
        // ソースの取り込み許可 CIDR
        whitelistCidr: '10.0.0.0/16',
        ingestPort: 5000,
      },
    });

    // 追加のアウトプット (RTP プッシュなど) は addOutput / addSource で追加可能
  }
}
```

**ポイント:**
- L2 の追加により、L1 (`CfnFlow`) を直接扱っていたコードは、より簡潔な API に移行できます。既存 L1 コードは互換性のためそのまま利用可能です。
- MediaConnect は RFC 4733 / SMPTE 2022-6 / SMPTE 2110 などの放送プロトコルに対応する伝送サービスであり、ライブ配信システムのソースとして利用されます。

### `core`: 合成テンプレートを既定ルールセットで自動検証 ([#38135](https://github.com/aws/aws-cdk/pull/38135))

**この機能でできること:**
- 新しいバリデーションプラグイン `CloudFormationValidatePlugin` が既定バリデーターとして追加されました。`@aws/cloudformation-validate` パッケージ (WASM ベースの CloudFormation Linter) をラップし、包括的なルールセットで合成後のテンプレートを検証します。
- デプロイ前の synth 段階でテンプレートの well-formedness (整形性) を検証し、問題を早期に検出できます。

**デフォルト動作:**

```typescript
// デフォルトでは、すべての診断結果は「警告」として報告され、synth は失敗しない
// v2.262.0 時点ではプラグインは新しく、warning 段階からの導入となっている
```

**厳格モードを有効化する例:**

```typescript
import { App } from 'aws-cdk-lib';

const app = new App({
  context: {
    // true にすると診断結果が「エラー」として分類され、
    // 違反がある場合は synth が失敗する
    '@aws-cdk/core:validateAgainstDefaultRules': true,
  },
});
```

**個別警告 / エラーの抑制:**

```typescript
import * as cdk from 'aws-cdk-lib';

cdk.Validations.of(scope).acknowledge({
  // 抑制する診断 ID
  id: 'CloudFormation-Validate::W9999',
  // 抑制の理由 (監査目的で必須)
  reason: 'This is not recommended but we have a good reason to do it like this',
});
```

**カスタムルール (Rego / CloudFormation Guard) の追加:**

```typescript
import * as cdk from 'aws-cdk-lib';

cdk.Validations.of(app).addPlugins(new cdk.CloudFormationValidatePlugin({
  guardRules: [
    'path/to/my/rules.guard',
  ],
}));
```

**ポイント:**
- 既定では警告レベルで報告されるため、既存プロジェクトの synth が急に失敗することはありません。フェールクローズドな導入が可能になった段階で、コンテキストキー `@aws-cdk/core:validateAgainstDefaultRules` を有効にしてください。
- 独自の Rego / CFN Guard ルールを追加することで、組織固有のガードレール (タグ必須、暗号化必須など) をチームで共有できます。
- 抑制 API `Validations.of(scope).acknowledge()` により、ID・理由をコード上に明示できます。

### `ecs`: Amazon ECS-optimized Amazon Linux 2023 (Neuron) AMI をサポート ([#34689](https://github.com/aws/aws-cdk/pull/34689))

**この機能でできること:**
- ECS の `EcsOptimizedImage` API で、Amazon Linux 2023 と Neuron ハードウェアの組み合わせ (`AL2023 + NEURON`) の AMI を解決できるようになりました。Inferentia / Trainium 系インスタンス (`inf2` など) を Amazon Linux 2023 系の ECS-optimized AMI で動作させられます。
- 従来は Amazon Linux 2 + Neuron のみサポートされており、SSM パラメータのパス命名規則が異なる Amazon Linux 2023 版は解決できませんでした。

**背景:**

AWS 公式ドキュメントによれば、Neuron AMI の SSM パラメータパスは以下のように異なります。

| AMI Variant | SSM Parameter Path |
|-------------|--------------------|
| Amazon Linux 2023 (Neuron) | `/aws/service/ecs/optimized-ami/amazon-linux-2023/neuron/<version>` |
| Amazon Linux 2 (Neuron) | `/aws/service/ecs/optimized-ami/amazon-linux-2/inf/<version>` |

**使用例:**

```typescript
import * as ec2 from 'aws-cdk-lib/aws-ec2';
import * as ecs from 'aws-cdk-lib/aws-ecs';

declare const vpc: ec2.Vpc;

const cluster = new ecs.Cluster(this, 'Cluster', { vpc });

// Amazon Linux 2023 (Neuron) の ECS-optimized AMI を利用
cluster.addCapacity('NeuronCapacity', {
  instanceType: new ec2.InstanceType('inf2.xlarge'),
  // amazonLinux2023() に HardwareType.NEURON を指定
  machineImage: ecs.EcsOptimizedImage.amazonLinux2023(
    ecs.AmiHardwareType.NEURON,
  ),
});
```

**ポイント:**
- Amazon Linux 2023 (Neuron) を利用したい場合は `EcsOptimizedImage.amazonLinux2023(AmiHardwareType.NEURON)` を指定します。
- Amazon Linux 2 (Neuron) を継続利用する場合は `EcsOptimizedImage.amazonLinux2(AmiHardwareType.NEURON)` が引き続き利用可能です。

### `cloudfront`: `Managed-HostHeaderOnly` オリジンリクエストポリシーを追加 ([#38236](https://github.com/aws/aws-cdk/pull/38236))

**この機能でできること:**
- CloudFront の新しいマネージドオリジンリクエストポリシー `Managed-HostHeaderOnly` を、`OriginRequestPolicy.HOST_HEADER_ONLY` として参照できるようになりました。
- このポリシーは Host ヘッダーのみをオリジンリクエストに含め、クエリ文字列や Cookie は含めません。オリジン側で Host ヘッダーだけを見れば十分なユースケース (シンプルなキャッシュキー設計) に適しています。

**使用例:**

```typescript
import * as cloudfront from 'aws-cdk-lib/aws-cloudfront';
import * as origins from 'aws-cdk-lib/aws-cloudfront-origins';

new cloudfront.Distribution(this, 'Distribution', {
  defaultBehavior: {
    origin: new origins.HttpOrigin('example.com'),
    // マネージドポリシーの Host ヘッダーのみを転送する設定を利用
    originRequestPolicy: cloudfront.OriginRequestPolicy.HOST_HEADER_ONLY,
  },
});
```

**ポイント:**
- 既存のマネージドポリシー (`ALL_VIEWER`, `CORS_S3_ORIGIN` など) と同じ形式で利用できます。
- カスタムポリシーを新規作成する必要がないため、簡潔にホストヘッダーのみ転送する設定を実現できます。

### `bedrockagentcore-alpha`: Gateway IAM 認証プロバイダーで `service` / `region` を指定可能に ([#37697](https://github.com/aws/aws-cdk/pull/37697))

**この機能でできること:**
- `@aws-cdk/aws-bedrock-agentcore-alpha` の `GatewayCredentialProvider.fromIamRole()` が、SigV4 署名時の `service` / `region` を指定できるようになりました。
- 従来はこれらのプロパティを L2 から設定できず、L1 (`CfnGatewayTarget`) の `IamCredentialProvider` を直接利用する必要がありました。

**使用例:**

```typescript
import * as agentcore from '@aws-cdk/aws-bedrock-agentcore-alpha';

// 従来通り引数なしでも動作 (後方互換)
const provider1 = agentcore.GatewayCredentialProvider.fromIamRole();

// service / region を指定して SigV4 の署名先を明示的に固定
const provider2 = agentcore.GatewayCredentialProvider.fromIamRole({
  // 署名対象のサービス (必須。IAM 認証プロバイダーブロックを使う場合)
  service: 'lambda',
  // 署名対象のリージョン (任意)
  region: 'us-east-1',
});
```

**ポイント:**
- `region` を指定する場合は `service` の指定が必須です。`region` のみ指定すると `ValidationError` が発生します (CFN 側の制約) 。
- 引数なしでの呼び出しは従来と同一の合成結果を生成するため、既存コードは影響を受けません。
- Alpha モジュールのため、API は今後変更される可能性があります。

## バグ修正

### `core`: `stack.node.addDependency` がスタック数の増加に伴い遅くなる問題 ([#38314](https://github.com/aws/aws-cdk/pull/38314))
- スタック間の依存関係を `sourceStack.node.addDependency(targetStack)` で設定する際、内部で両スタック配下のすべてのリソースをカルテシアン積で展開してから依存関係を適用していたため、スタックが大きくなるほど非線形に遅くなっていました。
- 修正後は、スタック間依存関係の場合は展開せず、直接スタックレベルで依存関係を適用するようになり、大幅に高速化されました。
- 副次的な変更として、内部 API 名がリネームされています (`stack.addDependency` → `stack.addStackDependency` 、`resource.addDependency` → `resource.addResourceDependency`) 。ユーザー向けの `construct.node.addDependency()` は変更ありません。

---

### `lambda`: バリデーションがトークン化された Provisioned Concurrency / 非同期呼び出し設定を拒否 ([#38246](https://github.com/aws/aws-cdk/pull/38246))
- `Alias` / `Version` の `provisionedConcurrentExecutions` 、`EventInvokeConfig` の `retryAttempts` / `maxEventAge` に `CfnParameter.valueAsNumber` などのトークン化された数値を渡すと、必ずバリデーションで拒否されていた問題を修正しました。
- CDK の数値トークンは内部的に大きな負の double 値としてエンコードされるため、下限チェックが常に真になっていました。修正後は `Token.isUnresolved()` / `Duration.isUnresolved()` で unresolved 値をスキップします。

**修正後の使用例:**

```typescript
import { CfnParameter } from 'aws-cdk-lib';
import * as lambda from 'aws-cdk-lib/aws-lambda';

declare const fn: lambda.Function;

const pce = new CfnParameter(this, 'PCE', { type: 'Number', default: 5 });

// CfnParameter からの deploy-time な値でも合成可能に
fn.addAlias('live', {
  provisionedConcurrentExecutions: pce.valueAsNumber,
});
```

### `backup`: `BackupPlanRule` / `BackupVault` の Lifecycle / VaultLock バリデーションがトークン化された Duration を拒否 ([#38264](https://github.com/aws/aws-cdk/pull/38264))
- `deleteAfter` / `moveToColdStorageAfter` の比較、`enableContinuousBackup` の 1-35 日レンジチェック、Vault Lock の `minRetention` / `maxRetention` / `changeableFor` の検証がトークン化された `Duration` を常に拒否していた問題を修正しました。
- `copyActions` にあった既存のトークンガードは `Token.isUnresolved(duration)` を使っていたため常に `false` を返し、実質無効化されていた点も修正されています。

---

### `elasticloadbalancingv2`: `dropInvalidHeaderFields` の `false` / 切り替え時の扱いを改善 ([#36483](https://github.com/aws/aws-cdk/pull/36483))
- ALB の `dropInvalidHeaderFields` を `false` に設定した場合、以前は CloudFormation テンプレートから属性自体が省略されていたため、CloudFormation のドリフト検出や変更セットでの検出ができませんでした。
- 修正後は `true` / `false` いずれも明示的に出力されるようになり、CloudFormation 側で正しく状態遷移を検出できます。

---

### `cloudfront-origins`: `HttpOrigin` の `httpPort` / `httpsPort` を範囲検証 ([#37872](https://github.com/aws/aws-cdk/pull/37872))
- `HttpOrigin` の `httpPort` / `httpsPort` にドキュメント外の値 (`0`, 負数, 65535 超, 小数など) を渡すと、synth をパスしてデプロイ時に汎用エラーになっていた問題を修正しました。
- 修正後は synth 時に `80`, `443`, または `1024-65535` のいずれかであることが検証されます。トークン値は `Token.isUnresolved()` でスルーされます。

### `certificatemanager`: `apexDomain` で `Object.prototype` メソッド名との衝突を防止 ([#37195](https://github.com/aws/aws-cdk/pull/37195))
- `apexDomain` ユーティリティが `in` 演算子でパブリックサフィックスマップを走査していたため、`toString` / `valueOf` などのドメインで誤ったドメインが返されていた問題を修正しました。
- 修正後は `Object.prototype.hasOwnProperty.call()` を利用し、自身のプロパティのみをチェックします。

### `opensearchservice`: gp3 EBS スループットの上限を 2000 MiB/s に引き上げ ([#38001](https://github.com/aws/aws-cdk/pull/38001))

### `core`: シンボリックリンク先がディレクトリの場合に synth が `EISDIR` でクラッシュ ([#38299](https://github.com/aws/aws-cdk/pull/38299))
- v2.258.0 以降、ポリシーバリデーションプラグインを登録した状態で `SymlinkFollowMode.NEVER` のアセットステージングや Docker バンドリング出力を含むクラウドアセンブリを扱うと、`hashFile()` が symlink 経由でディレクトリを読み込もうとして `EISDIR` エラーで synth がクラッシュしていた問題を修正しました。

### `validation`: 検証レポートの `templatePath` が cloud assembly に対して相対でない ([#38333](https://github.com/aws/aws-cdk/pull/38333), [#38352](https://github.com/aws/aws-cdk/pull/38352))
- バリデーションレポート内の `templatePath` が、CDK app のワーキングディレクトリからの相対パスや絶対パスなど不定でした。修正後は cloud assembly ディレクトリからの相対パスに統一され、cloud assembly が他マシンへ転送されてもパスが解決可能になっています。
- 併せて `stack.templateFullPath` が相対パスになり得るケースを、絶対パスに変換してから相対化する修正が入っています。

## Alpha モジュール

### `redshift-alpha`: 識別子と文字列リテラルの適切なエスケープ ([#38269](https://github.com/aws/aws-cdk/pull/38269))

**この機能でできること:**
- Redshift の Custom Resource ハンドラが SQL を組み立てる際、ユーザーやテーブル、カラム名などの識別子と、パスワードやコメントなどの文字列リテラルを適切にエスケープするようになりました。
- 識別子は特殊文字 (空白 / 引用符 / その他) を含む場合のみダブルクォートで囲まれ、埋め込まれたダブルクォートは倍加されます。文字列リテラルはシングルクォートで囲まれ、埋め込みのシングルクォートは倍加されます。

**ポイント:**
- 既に妥当な bare identifier となっている名前 (英数字とアンダースコアのみ) の出力は変わらないため、既存スタックは同一の Redshift オブジェクトに解決されます。
- 空白や引用符を含む名前が扱えるようになり、SQL の破壊や実行失敗を防げます。

## Revert / 未提供機能

以下 2 つの `feat` / `fix` は本リリースのリリースノートに記載されていますが、同一リリース内で revert されているため実質的には提供されていません。

### `apigateway`: ALB Integration ([#36247](https://github.com/aws/aws-cdk/pull/36247)) — [#38305](https://github.com/aws/aws-cdk/pull/38305) で revert
- API Gateway REST API から ALB へ統合する `AlbIntegration` が追加されましたが、`aws-apigateway` と `aws-apigatewayv2` の間で jsii Go バインディングに循環依存が発生し、パッケージング (`pack.sh`) がリリースパイプラインで失敗する状態でした。
- リリースをアンブロックするため、リリース直前に revert されています。ALB Integration は今後、`VpcLink` を共通の場所へ移す等、公開 API 上の循環参照を避ける形で再導入される予定です。

---

### `lambda`: Container Image Function での SnapStart サポート ([#38265](https://github.com/aws/aws-cdk/pull/38265)) — [#38281](https://github.com/aws/aws-cdk/pull/38281) で revert
- Lambda がプレビュー中のコンテナイメージ関数向け SnapStart を、CDK 側で許可する変更が入っていましたが、Lambda サービス側のアンゲート (一般提供化) が保留となったため revert されています。
- コンテナイメージ関数で SnapStart を利用したい場合は、Lambda サービス側での提供再開を待つ必要があります。

## まとめ

このリリースは、`aws-mediaconnect` の L2 追加と、`CloudFormationValidatePlugin` による合成テンプレートの既定バリデーションが大きな追加点です。加えて `stack.node.addDependency` のスタック間依存関係の性能改善や、Lambda / Backup のトークン化 Duration / 数値の扱いなど、実運用で刺さりやすいバグが多数修正されています。ALB Integration と SnapStart (Container Image) がリリースノートに含まれるものの revert されている点には注意してください。
