## 概要

このリリースでは、RDS の `Cluster` / `Instance` にマスターユーザーパスワードを Secrets Manager 側で自動生成・管理する `manageMasterUserPassword` プロパティが追加されました。また、合成される CloudFormation テンプレートに git リポジトリ URL と最新コミットハッシュを含む `AWS::CDK::Source` メタデータを付与できる機能や、L1 リソース定義の更新 (破壊的変更を含む) 、tarball アセットのシェルエスケープ修正などが含まれています。Alpha モジュールでは `metrics-facade-alpha` によるメトリクスファサードの自動生成が追加されました。

**リリース:** [v2.261.0](https://github.com/aws/aws-cdk/releases/tag/v2.261.0)

## 新機能

### `rds`: RDS Cluster / Instance で Secrets Manager ネイティブ統合をサポート ([#35734](https://github.com/aws/aws-cdk/pull/35734))

**この機能でできること:**
- `DatabaseCluster` / `DatabaseClusterFromSnapshot` / `DatabaseInstance` の props に `manageMasterUserPassword: true` を指定することで、RDS がマスターユーザーパスワードを Secrets Manager に自動生成・保存・ローテーションするネイティブ統合を有効化できます。
- CDK 側でパスワード用の Secret を作成せず、RDS が管理する Secret を `this.secret` として参照できるようになります。

**使用例:**

```typescript
import * as ec2 from 'aws-cdk-lib/aws-ec2';
import * as kms from 'aws-cdk-lib/aws-kms';
import * as rds from 'aws-cdk-lib/aws-rds';

declare const vpc: ec2.Vpc;
declare const kmsKey: kms.Key;

// Aurora Cluster — RDS がマスターパスワードを Secrets Manager で管理
new rds.DatabaseCluster(this, 'Cluster', {
  engine: rds.DatabaseClusterEngine.auroraMysql({
    version: rds.AuroraMysqlEngineVersion.VER_3_01_0,
  }),
  writer: rds.ClusterInstance.serverlessV2('writer'),
  vpc,
  // RDS ネイティブの Secrets Manager 統合を有効化 (デフォルト: false)
  manageMasterUserPassword: true,
  credentials: rds.Credentials.fromUsername('admin', {
    // Secret 暗号化用の KMS キー (省略時は AWS 管理のデフォルトキー)
    encryptionKey: kmsKey,
  }),
});

// RDS Instance でも同様に利用可能
new rds.DatabaseInstance(this, 'Instance', {
  engine: rds.DatabaseInstanceEngine.postgres({
    version: rds.PostgresEngineVersion.VER_16_3,
  }),
  vpc,
  manageMasterUserPassword: true,
  credentials: rds.Credentials.fromUsername('postgres'),
});
```

**ポイント:**
- `manageMasterUserPassword: true` の場合、`credentials` に指定できるのは `username` と `encryptionKey` のみです。`password` / `secret` / `secretName` / `excludeCharacters` / `replicaRegions` / `usernameAsString` を併用すると例外が発生します。
- `this.secret` は RDS 側で作成された Secret を `Secret.fromSecretAttributes()` で取り込んだ「参照」です。CDK 所有ではないため `addToResourcePolicy()` は no-op となります。アプリからアクセスを付与する際は `secret.grantRead(grantee)` を使用してください。
- `addRotationSingleUser()` / `addRotationMultiUser()` は併用不可 (`CannotAddRotationWithManageMasterUserPassword` 例外) 。ローテーションは RDS 側で自動的に行われます。
- Cluster の場合、`replicationSourceIdentifier` (リードレプリカクラスター) との併用も不可です (認証情報はソースクラスターから継承されるため) 。
- スナップショットからの復元 (`DatabaseClusterFromSnapshot`) にも対応しており、既存クラスターを RDS 管理パスワードへ移行するユースケースに使えます。

### `core`: CloudFormation テンプレートに git ソースメタデータを追加 ([#37368](https://github.com/aws/aws-cdk/pull/37368))

**この機能でできること:**
- 合成される CloudFormation テンプレートのトップレベル `Metadata` に、git リモートリポジトリ URL と最新コミットハッシュを含む `AWS::CDK::Source` エントリを追加できるようになりました。
- デプロイ済みのスタックがどのコミットから生成されたかを追跡でき、監査や再現性の担保に役立ちます。

**生成されるメタデータの例:**

```yaml
Metadata:
  AWS::CDK::Source:
    Repository: git@github.com:org/app.git
    Commit: 19e3e594a5512b75074526af89bf3b67a3164cc1
```

**有効化する使用例:**

```typescript
import { App } from 'aws-cdk-lib';

const app = new App({
  context: {
    // デフォルトは無効。true にすると git メタデータを埋め込む
    '@aws-cdk/core:enableGitSource': true,
  },
});
```

または `cdk.json` に以下を追加します。

```json
{
  "context": {
    "@aws-cdk/core:enableGitSource": true
  }
}
```

**ポイント:**
- デフォルトでは無効です。コンテキストキー `@aws-cdk/core:enableGitSource` を `true` に設定した場合のみ有効化されます。
- カレントディレクトリが git リポジトリでない場合や git コマンドが利用できない場合は、メタデータは静かに省略されます。
- セキュリティ観点から、`git` コマンドで取得したリポジトリ URL とコミットハッシュは想定フォーマットに合致する場合のみ利用されます。
- メタデータはクラウドアセンブリ (`cdk.out`) にも記録されます。

### L1 CloudFormation リソース定義の更新 ([#38189](https://github.com/aws/aws-cdk/pull/38189))

**この機能でできること:**
- `@aws-cdk/aws-service-spec` の最新変更を取り込み、L1 (`Cfn*`) リソース定義が広範に更新されました。
- 例として、`AWS::AmazonMQ::Broker` に `ResourceShareArns` が追加、`AWS::Bedrock::DataSource` に `AudioExtractionConfiguration` や `MANAGED_KNOWLEDGE_BASE_CONNECTOR` タイプが追加されるなど、多数のサービスで新規プロパティ / 属性 / 型が反映されています。

**ポイント:**
- L2 コンストラクトの API には基本的に影響しませんが、以下の 2 点は **破壊的変更** に分類されているため注意が必要です (「破壊的変更」節を参照) 。
- L1 を直接利用しているコードは、新しいプロパティを利用できるようになる一方、削除された属性への参照は修正が必要です。

## バグ修正

### 内部パス検出の改善 ([#38205](https://github.com/aws/aws-cdk/pull/38205))
- シンボリックリンクを扱う際、同じ接頭辞を持つ兄弟パスを内部パスと誤判定していた問題を修正しました。
- 例: `./hello/world/file.txt` に対して `./hello/world-something/new-dir/other-file.txt` が「`./hello/world/` の内部」と判定されていたケースが解消されます。
- 内部パス判定ロジックを 1 か所に集約したことで、今後のリグレッションも起きにくくなっています。

---

### `ecr-assets`: tarball アセットのビルドコマンドのシェルエスケープ ([#38171](https://github.com/aws/aws-cdk/pull/38171))
- `TarballImageAsset` のビルドコマンドで、tarball の相対パスがシェルスクリプト内に直接文字列連結されていた問題を修正しました。
- 修正前は `docker load -i <path> | tail -n 1 | sed "..."` のようにパスを直接展開していたため、パスに `$(...)` / `;` / バッククォートなどのシェルメタ文字が含まれると評価されるおそれがありました。
- 修正後は位置引数として渡す形に変更され、`"$1"` として展開されるためリテラルとして扱われます。

**修正後の内部呼び出し形式:**

```bash
sh -c 'docker load -i "$1" | tail -n 1 | sed "..."' cdk-tarball-image-asset <relativePathInOutDir>
```

**ポイント:**
- 通常のディレクトリ配下で tarball を扱っているユーザーには実害がない可能性が高いですが、パスにメタ文字を含めうる環境では改善効果があります。
- 挙動としては `executable` 配列の形が変わるため、内部的な synth 出力を assert しているコードがある場合は更新が必要です。

---

### 検証エラーを新しいスタイルで表示 ([#38166](https://github.com/aws/aws-cdk/pull/38166))
- synth 時の検証エラー (`Annotations.addWarning() / addError()` および policy validation プラグイン) を、CLI の `cdk validate` で導入された新しいフォーマッターで表示するようになりました。
- 従来のツリー型表示は廃止されています。

**主な挙動 (context キー別):**

| `failSynthOnValidationErrors` | `annotationsInValidationReport` | 動作 |
|---|---|---|
| `true` / 未設定 | `true` | すべての annotation (warning / error) を表示し、error があれば exit |
| `true` / 未設定 | `false` | policy validation プラグインの結果のみ表示し、error があれば exit |
| `false` | - | 何も表示しない |

**ポイント:**
- annotation から変換された error は「suppress 可能」として扱われなくなりました (`fatal` でなくても suppress 不可) 。
- ルール名 / 修正提案 (suggested fix) / プラグイン失敗の表示が改善され、ファイルパスは現在の作業ディレクトリからの相対パスで描画されます。
- construct パスが取れない場合は空文字列で描画されます (以前は `"N/A"`) 。

## Alpha モジュール

### `metrics-facade-alpha`: メトリクスファサードの自動生成 ([#37334](https://github.com/aws/aws-cdk/pull/37334))

**この機能でできること:**
- 各 L1 / L2 リソースに対応した「メトリクスファサード」クラスを自動生成する仕組みが追加されました。
- L2 のリソース参照 (`IFunctionRef` など) からファクトリメソッド経由でメトリクスを取得でき、ディメンションが自動的に埋められます。

**使用例 (Lambda):**

```typescript
import * as lambda from 'aws-cdk-lib/aws-lambda';
import * as cloudwatch from 'aws-cdk-lib/aws-cloudwatch';
import { LambdaMetrics } from '@aws-cdk/metrics-facade-alpha';

declare const fn: lambda.Function;

// L2 リソースからスコープされたメトリクスファサードを生成
// (内部で FunctionName ディメンションが自動的にセットされる)
const fnMetrics = LambdaMetrics.fromFunction(fn);

// メトリクスを取得
const errorMetric = fnMetrics.metricErrors();

// 通常どおり Alarm 等に利用可能
new cloudwatch.Alarm(this, 'L2ErrorAlarm', {
  metric: errorMetric,
  alarmName: 'MyApp-Lambda-Errors',
  threshold: 5,
  evaluationPeriods: 1,
  comparisonOperator: cloudwatch.ComparisonOperator.GREATER_THAN_THRESHOLD,
});
```

**生成されるクラス構造 (Lambda の例、抜粋):**

```typescript
export class LambdaMetrics {
  // Function 参照からメトリクスを作成 (ディメンションを自動補完)
  static fromFunction(ref: IFunctionRef): LambdaMetrics.FunctionMetrics;
}

export namespace LambdaMetrics {
  // 関数単位のメトリクス (dimension: FunctionName)
  class FunctionMetrics {
    constructor(props: { functionName: string });
    metricInvocations(options?: MetricOptions): IMetric;
    metricErrors(options?: MetricOptions): IMetric;
    metricDuration(options?: MetricOptions): IMetric;
    metricThrottles(options?: MetricOptions): IMetric;
    // ...
  }

  // 関数 + リソース単位 (dimensions: FunctionName, Resource)
  class ResourceMetrics { /* ... */ }

  // バージョン単位 (dimensions: ExecutedVersion, FunctionName, Resource)
  class ExecutedVersionMetrics { /* ... */ }

  // アカウント全体 (dimensions なし)
  class AccountMetrics { /* ... */ }
}
```

**ポイント:**
- Alpha モジュール (`@aws-cdk/metrics-facade-alpha`) のため、API は今後変更される可能性があります。
- 各リソースの Metrics クラスは自動生成されるため、以前は手動で `new cloudwatch.Metric({ ... })` を書いていた箇所を型付きの API で置き換えられます。

## 破壊的変更

L1 リソースは公開 CloudFormation Resource Schema から自動生成されるため、まれに以前の型と互換性のない変更が入ることがあります。今回のリリースでは以下 2 件の破壊的変更が含まれています ([#38189](https://github.com/aws/aws-cdk/pull/38189)) 。

### `aws-cloudwatch`: `AWS::CloudWatch::LogAlarm` の `QueryLanguage` プロパティ削除

**変更内容:**
- `ScheduledQueryConfiguration` 型から `QueryLanguage` プロパティが削除されました。

**移行方法:**
- L1 (`CfnLogAlarm`) の `scheduledQueryConfiguration` で `queryLanguage` を指定していた場合は、指定を削除してください。

---

### `aws-elasticloadbalancing`: `AWS::ElasticLoadBalancing::LoadBalancer` の `Id` 属性削除

**変更内容:**
- 主要識別子 (primary identifier) が `Id` 属性から `LoadBalancerName` に変更されました。`Id` 属性は削除されています。

**変更前:**

```typescript
const loadBalancerId = cfnLoadBalancer.attrId;
```

**変更後:**

```typescript
// LoadBalancerName を主要識別子として利用する
const loadBalancerName = cfnLoadBalancer.loadBalancerName;
```

**移行方法:**
- L1 (`CfnLoadBalancer`) で `attrId` を参照しているコードは、`loadBalancerName` プロパティ (または新しい主要識別子) を利用するよう修正してください。
- L2 の `elbv1.LoadBalancer` を利用している場合は影響しません。

## まとめ

このリリースの目玉は、RDS Cluster / Instance の Secrets Manager ネイティブ統合と、CloudFormation テンプレートへの git ソースメタデータ埋め込みです。加えて L1 更新に伴う破壊的変更 (CloudWatch LogAlarm / ELB Classic) が含まれるため、該当する L1 を利用している場合はアップグレード前に確認してください。
