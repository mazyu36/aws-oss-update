## 概要

このリリースでは、S3 の account-regional 名前空間 (`bucketNamePrefix` / `bucketNamespace`) サポート、検証レポートがデフォルトで Cloud Assembly に書き出されるようになる改善、および EC2 `IPeer` メソッドの戻り値の型安全性向上が追加されました。また、大規模な Construct ツリーでスタックオーバーフローを引き起こしていた問題と、トークンでラップされた `Box` 値のマージ不具合が修正されています。

**リリース:** [v2.256.0](https://github.com/aws/aws-cdk/releases/tag/v2.256.0)

---

## 新機能

### S3 Bucket で `bucketNamePrefix` と `bucketNamespace` をサポート ([#37386](https://github.com/aws/aws-cdk/pull/37386))

**この機能でできること:**
- L1 `AWS::S3::Bucket` リソースに追加された `BucketNamePrefix` / `BucketNamespace` プロパティが、L2 `Bucket` Construct から利用できるようになりました。
- アカウント・リージョンスコープ (account-regional namespace) でバケットを作成できるようになり、グローバルに一意な名前を考える必要がなく、他アカウントにバケット名を奪われるリスクも回避できます。

**使用例:**

```typescript
import * as s3 from 'aws-cdk-lib/aws-s3';

// account-regional 名前空間のバケット
// 実際のバケット名は <prefix>-<accountId>-<region>-an の形式で作成される
const bucket = new s3.Bucket(this, 'MyBucket', {
  bucketNamePrefix: 'my-app-data',                  // 接頭辞 (37 文字以下、小文字英数とハイフン)
  bucketNamespace: s3.BucketNamespace.ACCOUNT_REGIONAL, // GLOBAL (デフォルト) / ACCOUNT_REGIONAL
});

// グローバル名前空間 (従来どおり)
const legacyBucket = new s3.Bucket(this, 'LegacyBucket', {
  bucketName: 'my-globally-unique-bucket-name',
  // bucketNamespace のデフォルトは GLOBAL
});
```

**ポイント:**
- `BucketNamespace` enum には `GLOBAL` と `ACCOUNT_REGIONAL` の 2 値が用意されています。
- `bucketNamePrefix` は **小文字英数字とハイフンのみ**、**先頭・末尾は英字または数字**、**37 文字以下** という制約があります (account-regional のサフィックス分の余裕を考慮)。
- 以下のプロパティの組み合わせは Construct 時にエラーになります:
  - `bucketName` と `bucketNamePrefix` の併用
  - `bucketName` と `bucketNamespace`(`GLOBAL` 以外) の併用
  - `bucketNamePrefix` を `ACCOUNT_REGIONAL` 以外で使用
  - `bucketNamespace: ACCOUNT_REGIONAL` を `bucketNamePrefix` なしで使用
- account-regional バケットでは、CloudFormation 側の自動生成名がサフィックス要件を満たさないため、`bucketNamePrefix` の明示的な指定が必須となります。

### 検証レポートが常に Cloud Assembly に書き出されるように ([#37867](https://github.com/aws/aws-cdk/pull/37867))

**この機能でできること:**
- ポリシー検証 (Policy Validation) のレポートが、明示的な設定なしで Cloud Assembly ディレクトリに `policy-validation-report.json` として書き出されるようになりました。
- 同時に、stderr への pretty-printed 形式での出力もデフォルトで有効になります。
- これにより、CDK CLI の `validate` コマンドが追加設定なしで検証レポートを取得できるようになりました。

**使用例 (出力を抑止する場合):**

```json
// cdk.json
{
  "context": {
    "@aws-cdk/core:validationReportJson": false,        // JSON ファイル出力を抑止
    "@aws-cdk/core:validationReportPrettyPrint": false  // stderr へのコンソール出力を抑止
  }
}
```

**ポイント:**
- 従来は `@aws-cdk/core:validationReportJson` を `true` に明示しないと JSON ファイルが書き出されませんでしたが、今回からデフォルトで `true` です。
- 2 つの出力フォーマット (JSON / pretty-print) はそれぞれ独立して `false` に設定することで個別に無効化できます。
- これは挙動変更ですが、機能の追加であり、既存ファイルが上書きされることはありません。

### `IPeer` メソッドの戻り値型を具体的なインターフェースに ([#36637](https://github.com/aws/aws-cdk/pull/36637))

**この機能でできること:**
- `IPeer` インターフェースの `toIngressRuleConfig()` / `toEgressRuleConfig()` メソッドの戻り値が、これまでの `any` から具体的なインターフェース (`IngressRuleConfig` / `EgressRuleConfig`) に変更されました。
- TypeScript 利用時の型補完・型チェックが効くようになり、ランタイムエラーの早期発見につながります。

**使用例:**

```typescript
import * as ec2 from 'aws-cdk-lib/aws-ec2';

const peer = ec2.Peer.ipv4('10.0.0.0/16');

// 戻り値が IngressRuleConfig 型になり、IDE で補完が効く
const ingressConfig: ec2.IngressRuleConfig = peer.toIngressRuleConfig();
//   cidrIp?: string;
//   cidrIpv6?: string;
//   sourcePrefixListId?: string;
//   sourceSecurityGroupId?: string;
//   sourceSecurityGroupOwnerId?: string;

const egressConfig: ec2.EgressRuleConfig = peer.toEgressRuleConfig();
//   cidrIp?: string;
//   cidrIpv6?: string;
//   destinationPrefixListId?: string;
//   destinationSecurityGroupId?: string;
```

**ポイント:**
- `any` から具体型への絞り込みのため後方互換性は保たれており、既存コードの修正は不要です。
- 内部の `CidrIPv4` / `CidrIPv6` / `PrefixList` / `SecurityGroupId` などの `Peer` 実装、および `SecurityGroupBase` / `PrefixListBase` も同インターフェースを返すよう更新されています。

### 合成が遅い場合にパフォーマンスカウンタを出力 ([#37919](https://github.com/aws/aws-cdk/pull/37919))

**この機能でできること:**
- 合成 (synthesis) 処理がしきい値 (1 スタックあたり 10 秒) を超えた場合に、フレームワークがパフォーマンスカウンタをファイルに書き出すようになりました。
- CLI はこのファイルをテレメトリとして送信し、CDK チームがボトルネック分析に活用します。
- v2.255.0 で取り込まれた [#37843](https://github.com/aws/aws-cdk/pull/37843) の re-roll であり、`fs-extra` のモンキーパッチ警告に対応した修正版が改めて取り込まれました。

**使用例 (レポーティングの無効化):**

```typescript
import { App } from 'aws-cdk-lib';

// App プロパティで無効化
const app = new App({
  performanceReporting: false,
});
```

```json
// または cdk.json で
{
  "context": {
    "aws:cdk:performance-reporting": false
  }
}
```

**ポイント:**
- 出力先のファイルパスは `$CDK_PERF_COUNTERS_FILE` 環境変数で指定されます。
- 出力されるカウンタの例: `phase:Construction`、`bundle:NodejsFunction`、`DockerImage.fromBuild`、`FileSystem.fingerprint` など、各処理の所要時間 (ms) と呼び出し回数。
- ユーザーは `printPerfCounters()` 関数を使って独自のプロファイリングにも活用できます。

## バグ修正

### 大規模な Construct ツリーでのスタックオーバーフローを修正 ([#37901](https://github.com/aws/aws-cdk/pull/37901))

- 数十万ノード規模の巨大な CDK アプリで、合成中に `RangeError: Maximum call stack size exceeded` が発生する問題が修正されました。
- 原因は `tree.json` メタデータ生成時の `nodeCount` 関数が深い再帰でツリーサイズを計算しており、V8 のコールスタックを使い切っていたことでした。
- 今回の修正では再帰実装をスタックベースの反復実装に置き換え、`O(N)` 時間で任意の深さ・幅の Construct ツリーを安全にトラバースできるようになっています。
- 生成される CloudFormation テンプレートには影響しません。

---

### トークンでラップされた `Box` をプロパティマージ戦略で正しく扱うように修正 ([#37902](https://github.com/aws/aws-cdk/pull/37902))

- `BoxSafeMergeStrategy` および `BoxSafeArrayStrategy` がこれまで `Box.isBox()` による直接的な Box インスタンス検出のみに頼っていたため、`Lazy.any()` や `Lazy.string()` で Box がトークンにエンコードされたケースを取り逃していました。
- これにより、トークンが単なる値として扱われ、プロパティの遅延マージ (deferred merge) が適切に行われずに誤った結果になることがありました。
- 修正では `Tokenization.reverse()` を使ってトークンエンコードされた Box を取り出す `tryUnbox()` ヘルパが追加され、両戦略で利用されるようになりました。
- v2.255.0 で導入された [#37844](https://github.com/aws/aws-cdk/pull/37844) のフォローアップ的な修正です。

## まとめ

S3 の account-regional 名前空間サポートと検証レポートのデフォルト有効化は、運用面で実質的な改善をもたらす嬉しい変更です。あわせて、巨大 Construct ツリーでのスタックオーバーフロー修正など重要な信頼性改善も含まれており、大規模アプリケーションの利用者は積極的にアップデートを検討してください。
