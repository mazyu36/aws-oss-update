## 概要

このリリースは、Lambda の `Runtime.NODEJS_LATEST` が `nodejs24.x` に更新される破壊的変更を含みます。新機能として ECS Service の強制再デプロイオプション、EKS Managed Node Group の AL2023 デフォルト化 (Feature Flag 配下) 、cross-stack 参照を弱参照に誘導する変更などが含まれます。

**リリース:** [v2.259.0](https://github.com/aws/aws-cdk/releases/tag/v2.259.0)

---

## 新機能

### ECS Service に `forceNewDeployment` オプションを追加 ([#36797](https://github.com/aws/aws-cdk/pull/36797))

**この機能でできること:**
- `FargateService` および `Ec2Service` で、CloudFormation の `ForceNewDeployment` プロパティが利用可能になりました。
- サービス定義に変更がない場合でも、新しいデプロイを強制的にトリガーできるようになります。

**使用例:**

```typescript
import * as ecs from 'aws-cdk-lib/aws-ecs';

new ecs.FargateService(this, 'Service', {
  cluster,
  taskDefinition,
  forceNewDeployment: {
    enabled: true,                    // 強制再デプロイの有効化 (forceNewDeployment 指定時のデフォルト: true)
    nonce: Date.now().toString(),     // 任意の値 (例: タイムスタンプやコミットハッシュ) 。値を変更すると新しいデプロイがトリガーされる
  },
});
```

**ポイント:**
- `latest` タグを使った Docker イメージの更新時、新しい Fargate プラットフォームバージョンへのロール時など、サービスパラメータを変更せずに再デプロイしたい場合に有用です。
- `enabled` を省略して `nonce` のみ指定した場合は `enabled: true` として扱われます。
- `Ec2Service` でも同じプロパティが利用できます。

### EKS Managed Node Group のデフォルト AMI を AL2023 に変更 (Feature Flag) ([#37850](https://github.com/aws/aws-cdk/pull/37850))

**この機能でできること:**
- 新しい Feature Flag `@aws-cdk/aws-eks:defaultToAL2023` が追加され、有効化すると `eks.Cluster` (`defaultCapacity > 0`) や `cluster.addNodegroupCapacity()` で `amiType` を指定しなかった場合に、デフォルトの AMI Type が AL2 (`AL2_x86_64` / `AL2_ARM_64`) から AL2023 (`AL2023_X86_64_STANDARD` / `AL2023_ARM_64_STANDARD`) に変わります。
- Amazon Linux 2 EKS-optimized AMI のサポートが 2025 年 11 月 26 日に終了したことを受けた変更です。

**使用例:**

```typescript
import * as eks from 'aws-cdk-lib/aws-eks';
import { KubernetesVersion } from 'aws-cdk-lib/aws-eks';

// cdk.json で feature flag を有効化
// "@aws-cdk/aws-eks:defaultToAL2023": true

new eks.Cluster(this, 'MyCluster', {
  version: KubernetesVersion.V1_31,
  defaultCapacity: 2,
  // amiType を省略すると、Feature Flag 有効時は AL2023_X86_64_STANDARD が使われる
});
```

**ポイント:**
- **既存アプリで Feature Flag を有効化する際の注意点**: 既存の Managed Node Group がすべて置換され、稼働中の Pod が終了します。アップグレード前に既存の Node Group すべてに `amiType: NodegroupAmiType.AL2_X86_64` (または `AL2_ARM_64`) を明示的に指定してください。これにより、Feature Flag 有効化はノーオペとなり、Node Group ごとに段階的に移行できます。
- GPU インスタンスは引き続き `AL2_X86_64_GPU` がデフォルトのままです。AL2023 では GPU サポートが `AL2023_X86_64_NVIDIA` と `AL2023_X86_64_NEURON` に分かれており、CDK 側で自動判別できないためです。
- セルフマネージド ASG (`EksOptimizedImage`) は本変更の対象外です。
- `aws-eks` および `aws-eks-v2` の両方で同様に動作します。

### `core`: cross-stack 参照のデフォルト推奨を「弱参照」に変更 ([#38070](https://github.com/aws/aws-cdk/pull/38070))

**この機能でできること:**
- `@aws-cdk/core:defaultCrossStackReferences` Feature Flag の **推奨値**が `weak` に変更されました (デフォルト動作は後方互換のため `strong` のまま) 。
- 値が未設定の場合、`cdk synth` 時に「設定されていない feature flag があります」という警告が表示され、`cdk flags --unstable=flags` で詳細を確認できます。
- `weak` を明示的に設定するか、`strong` を明示的に設定すれば警告は出なくなります。

**移行ガイダンス:**
- **未設定**: `both` に設定して全環境にデプロイすることを推奨する警告が表示されます。
- **`both` 設定済み**: 全環境にデプロイ完了後、`weak` に切り替えることを推奨する警告が表示されます。
- **`strong` または `weak` 設定済み**: 警告は表示されません (ユーザーの明示的な選択を尊重) 。

**ポイント:**
- 弱参照を使うと、CloudFormation の Export/Import を介さずに値がインライン展開され、cross-stack 間の "deadly embrace" (相互ロック) を回避できます。
- v2.258.0 で追加された `CrossStackReferences.of(...).consume(...)` API と組み合わせて、scope 単位 / リソース単位 / 利用箇所単位での細かい制御も可能です。

## バグ修正

### `spec2cdk`: EventBridge イベント名前空間名のハイフンをサニタイズ ([#38088](https://github.com/aws/aws-cdk/pull/38088))
- `aws.account@RegionOpt-InStatusChange` のようにハイフンを含む新しいスキーマ名がコード生成時にエラーになっていた問題を修正しました。
- ハイフンは削除され、後続の文字が大文字化される (camelCase) 形で正規化され、コード生成が正常に完了するようになりました。
- スキーマ名にハイフンを含む新しい EventBridge イベント名前空間が L1 リソースとして利用できるようになります。

## 破壊的変更

### `lambda`: `Runtime.NODEJS_LATEST` が `nodejs24.x` に更新 ([#38031](https://github.com/aws/aws-cdk/pull/38031))

Node.js 24 が 2025 年 11 月に AWS Lambda で GA になり、AWS の現行推奨 LTS となったことを受け、`Runtime.NODEJS_LATEST` が全リージョンで `nodejs24.x` を返すようになりました。

**影響範囲:**
- `Runtime.NODEJS_LATEST` を直接利用している Lambda 関数。
- フレームワーク内部で `determineLatestNodeRuntimeName` を経由するシングルトン (`LogRetention`、`CustomResourceProvider`、`S3 auto-delete-objects`、`dns-validated certificate` など) 。

**変更前:**

```typescript
import * as lambda from 'aws-cdk-lib/aws-lambda';

new lambda.Function(this, 'Fn', {
  runtime: lambda.Runtime.NODEJS_LATEST,  // → nodejs22.x
  handler: 'index.handler',
  code: lambda.Code.fromAsset('lambda'),
});
```

**変更後:**

```typescript
import * as lambda from 'aws-cdk-lib/aws-lambda';

new lambda.Function(this, 'Fn', {
  runtime: lambda.Runtime.NODEJS_LATEST,  // → nodejs24.x
  handler: 'index.handler',
  code: lambda.Code.fromAsset('lambda'),
});
```

**移行方法:**

1. **`Runtime.NODEJS_LATEST` を引き続き使う場合 (推奨):**
   - 既存の `AWS::Lambda::Function` リソースは次回デプロイ時に `Runtime: nodejs22.x` から `Runtime: nodejs24.x` にインプレースで更新されます。
   - **重要**: Node.js 24 ではコールバック形式の非同期ハンドラ (`(event, context, callback) => {...}`) のサポートが廃止されています。コールバック形式を使用しているコードは `async (event, context) => {...}` 形式に移行してください。

2. **`nodejs22.x` に固定したい場合:**

   ```typescript
   new lambda.Function(this, 'Fn', {
     runtime: lambda.Runtime.NODEJS_22_X,  // 明示的に固定
     // ...
   });
   ```

3. **`aws-lambda-nodejs` の `NodejsFunction` で固定したい場合:**

   ```typescript
   import * as lambdaNodejs from 'aws-cdk-lib/aws-lambda-nodejs';

   new lambdaNodejs.NodejsFunction(this, 'Fn', {
     runtime: lambda.Runtime.NODEJS_22_X,
     useLatestRuntimeVersion: false,  // 明示的に固定
     // ...
   });
   ```

**ポイント:**
- 具体的なランタイムバージョンを指定 (`Runtime.NODEJS_22_X` など) しているコードは影響を受けません。
- Lambda はランタイムバージョンの更新をインプレースで受け付けるため、CloudFormation 上はリソース置換にはなりません。
- 移行前にコールバック形式ハンドラの使用有無を必ず確認してください。

## まとめ

このリリースは、Node.js 24 への Lambda デフォルトランタイム更新という重要な破壊的変更を含みます。コールバック形式ハンドラを使用している場合は、`async/await` 形式への移行または `NODEJS_22_X` への明示的な固定が必要です。EKS の AL2023 デフォルト化 (Feature Flag) や ECS Service の `forceNewDeployment` など、運用面で有用な機能も追加されています。
