## 概要

このリリースは、ポリシー検証 (Policy Validation) フレームワークの大幅な拡張、cross-stack 参照の強度を細かく制御できる新 API、EKS Cluster の削除保護プロパティ、ElastiCache の enum-like class への移行などを含む大型リリースです。L1 リソース定義の更新に伴い、`aws-pcaconnectorad` で複数の必須プロパティ化という破壊的変更が含まれます。

**リリース:** [v2.258.0](https://github.com/aws/aws-cdk/releases/tag/v2.258.0)

## 新機能

### EKS Cluster に `deletionProtection` プロパティを追加 ([#36474](https://github.com/aws/aws-cdk/pull/36474))

**この機能でできること:**
- EKS Cluster の意図しない削除を防ぐための削除保護を、CDK 側で直接設定できるようになりました。
- 従来は CloudFormation の Escape Hatch を使う必要がありましたが、`ClusterProps` の標準プロパティとして公開されました。

**使用例:**

```typescript
import * as eks from 'aws-cdk-lib/aws-eks';
import { KubernetesVersion } from 'aws-cdk-lib/aws-eks';

new eks.Cluster(this, 'MyCluster', {
  version: KubernetesVersion.V1_31,
  deletionProtection: true,  // 削除保護を有効化 (デフォルト: false)
});
```

**ポイント:**
- `deletionProtection: true` を設定すると、CloudFormation 経由でクラスターを削除しようとした際にエラーが返るようになります。
- 本番環境のクラスターなど、誤削除が業務に影響を与える可能性がある場合に推奨されます。

### EKS の AlbControllerVersion に v2.8.3 〜 v3.2.2 を追加 ([#37752](https://github.com/aws/aws-cdk/pull/37752))

**この機能でできること:**
- AWS Load Balancer Controller の v2.8.3 〜 v3.2.2 (24 バージョン) が `AlbControllerVersion` の static 定数として追加されました。
- 従来 `AlbControllerVersion.of()` Escape Hatch でバージョンと IAM ポリシーを手動管理する必要がありましたが、これらの新バージョンは組み込みでサポートされます。

**使用例:**

```typescript
import * as eks from 'aws-cdk-lib/aws-eks';
import { KubernetesVersion } from 'aws-cdk-lib/aws-eks';

const cluster = new eks.Cluster(this, 'MyCluster', {
  version: KubernetesVersion.V1_31,
  albController: {
    // v2.8.3 〜 V3_2_2 まで static 定数として利用可能
    version: eks.AlbControllerVersion.V3_2_2,
  },
});
```

**ポイント:**
- v3.x は v2.x と IAM ポリシーが異なります。CDK 内に各バージョン対応のポリシー JSON が同梱されているので、追加設定は不要です。
- `aws-eks` と `aws-eks-v2` の両モジュールで同じバージョンが利用可能です。

### ElastiCache の `CacheEngine` / `UserEngine` を enum-like class へ変更 ([#37816](https://github.com/aws/aws-cdk/pull/37816))

**この機能でできること:**
- alpha モジュール `@aws-cdk/aws-elasticache-alpha` の `CacheEngine` と `UserEngine` が、TypeScript enum から enum-like class に変更されました。
- これにより、CDK のリリースを待たずに新しいエンジンバージョン (例: 2026 年 5 月リリースの Valkey 9.0 for ElastiCache Serverless) を `.of()` ファクトリで指定できるようになります。
- 同時に `CacheEngine.VALKEY_9` も追加されました。

**使用例:**

```typescript
import * as elasticache from '@aws-cdk/aws-elasticache-alpha';

// 既存の使い方は変わらず、定数アクセスがそのまま使える
new elasticache.ServerlessCache(this, 'Cache1', {
  engine: elasticache.CacheEngine.VALKEY_8,
});

// 新たに追加された Valkey 9
new elasticache.ServerlessCache(this, 'Cache2', {
  engine: elasticache.CacheEngine.VALKEY_9,
});

// CDK にまだ定数がない新しいエンジンバージョンを指定する場合
new elasticache.ServerlessCache(this, 'Cache3', {
  engine: elasticache.CacheEngine.of('valkey', '9'),
});
```

**ポイント:**
- alpha モジュールの破壊的変更: 値を文字列として比較していたコードは `.engineType` / `.majorEngineVersion` プロパティ経由のアクセスに書き換える必要があります (props として渡す用法は変わりません)。
- enum-like class パターン (PostgresEngineVersion などと同じ) への移行で、将来のエンジンバージョン追加を CDK のリリースを待たずに利用可能になります。
- `CacheEngine.MEMCACHED_1_6` とユーザーグループ併用時の検証バグも併せて修正されています。

### Bedrock AgentCore Runtime のアプリケーションロググループを公開 ([#37812](https://github.com/aws/aws-cdk/pull/37812))

**この機能でできること:**
- `aws-bedrockagentcore` の `Runtime` L2 構造に、デフォルトエンドポイントの AgentCore マネージドアプリケーションロググループを参照する `applicationLogGroup` プロパティが追加されました。
- 従来はロググループ名を文字列で組み立てて `LogGroup.fromLogGroupName()` を呼ぶ必要がありましたが、`runtime.applicationLogGroup` 経由で型安全に取得できます。

**使用例:**

```typescript
import * as bedrockagentcore from 'aws-cdk-lib/aws-bedrockagentcore';
import * as logs from 'aws-cdk-lib/aws-logs';
import * as cw from 'aws-cdk-lib/aws-cloudwatch';

const runtime = new bedrockagentcore.Runtime(this, 'MyRuntime', {
  // ...
});

// 自動的に /aws/bedrock-agentcore/runtimes/{agentRuntimeId}-DEFAULT を参照
const metricFilter = new logs.MetricFilter(this, 'ErrorFilter', {
  logGroup: runtime.applicationLogGroup,
  filterPattern: logs.FilterPattern.literal('ERROR'),
  metricNamespace: 'MyApp',
  metricName: 'AgentCoreErrors',
});

new cw.Alarm(this, 'ErrorAlarm', {
  metric: metricFilter.metric(),
  threshold: 1,
  evaluationPeriods: 1,
});
```

**ポイント:**
- 取得は遅延評価 (Lazy) され、`applicationLogGroup` を一度も参照しない Runtime には construct tree に余計なノードが追加されません。
- `Runtime.fromAgentRuntimeAttributes()` でインポートした Runtime でも同じプロパティが利用可能です。
- ロググループパスのハードコード抜けによる「アラームが発火しない」事故を防げます。

### Cross-stack 参照の強度を細かく制御する新 API ([#37840](https://github.com/aws/aws-cdk/pull/37840))

**この機能でできること:**
- CDK の cross-stack 参照 (リソースを別スタックから参照したときに発生する CloudFormation の `Export`/`ImportValue`) は、デフォルトで「強参照」(`strong`) になり、消費側の状態次第でリソース更新時に "deadly embrace" (相互ロック) が発生することがあります。
- 既存の context key `@aws-cdk/core:defaultCrossStackReferences` はアプリ全体に対する all-or-nothing の制御でしたが、本リリースで **scope 単位 / リソース単位 / 利用箇所単位** の 3 階層で参照強度を細かく制御できる API が追加されました。

**使用例:**

```typescript
import { CrossStackReferences, ReferenceStrength } from 'aws-cdk-lib';

// 1. リソース単位 (Producer 側): このリソースへの全参照を弱参照に
CrossStackReferences.of(bucket).produce(ReferenceStrength.WEAK);

// 2. 利用箇所単位 (Consumer 側): 単一の参照のみ弱参照に
consumerStack.consumeReference(bucket.bucketArn, ReferenceStrength.WEAK);

// 3. scope 単位 (Consumer 側): scope 配下のすべての参照を弱参照に
CrossStackReferences.of(consumerStack).consume(ReferenceStrength.WEAK);
```

**ポイント:**
- 解決優先度: **利用箇所単位 > リソース単位 > scope context > デフォルト (`strong`)**
- "deadly embrace" 問題が発生したリソースだけをピンポイントに弱参照化できるため、アプリ全体の挙動を変えることなく特定の問題を解消できます。
- weak 参照では、CloudFormation Export ではなく値を直接インライン展開する形になり、依存関係の循環を解消できます (代わりに参照側の自動更新は行われません)。

### `core` のポリシー検証フレームワーク強化

#### Validation Report の新スキーマを追加 ([#37970](https://github.com/aws/aws-cdk/pull/37970))

**この機能でできること:**
- CDK CLI の `validate` コマンドが利用する新しい validation report スキーマに対応しました。
- 新しいレポート (`validation-report.json`) はトップレベルに `version` を持ち、`pluginName` / `conclusion` / `severity` (typed enum: `fatal | error | warning | info | custom`) / `violatingConstructs` (`constructFqn`、`libraryVersion`、`cloudFormationResource`、`stackTraces` を含む) / `suggestedFix` などを持ちます。
- 旧フォーマット (`policy-validation-report.json`) は context key `@aws-cdk/core:validationReportJson` を `true` に設定した場合のみ書き出されます。

**ポイント:**
- 旧 CLI との互換性のため、レガシーレポートはオプトインで利用可能です。
- `cx-api` から `VALIDATION_REPORT_FILE`、`LEGACY_VALIDATION_REPORT_FILE`、`VALIDATION_REPORT_JSON_CONTEXT` 定数が公開されています。

---

#### `IPolicyValidationContext` に scope を追加 ([#38006](https://github.com/aws/aws-cdk/pull/38006))

**この機能でできること:**
- ポリシー検証プラグインが受け取る `IPolicyValidationContext` (および非推奨の `IPolicyValidationContextBeta1`) に、検証対象アプリのルート construct を表す `scope` プロパティが追加されました。
- プラグインは、合成済み JSON テンプレートを再パースする代わりに construct tree を直接走査して、L1 プロパティへ型付きアクセスしたり `Stack.of(node).resolve()` でトークン解決したりできるようになります。

**使用例 (プラグイン内):**

```typescript
import {
  IPolicyValidationPluginBeta1,
  IPolicyValidationContext,
  PolicyValidationPluginReportBeta1,
} from 'aws-cdk-lib';

class MyPlugin implements IPolicyValidationPluginBeta1 {
  readonly name = 'MyPlugin';

  validate(context: IPolicyValidationContext): PolicyValidationPluginReportBeta1 {
    // 新しい scope プロパティ経由でルート construct にアクセスできる
    const root = context.scope;
    if (root) {
      // construct tree を走査して L1 プロパティへ直接アクセス
      // root.node.findAll().forEach(...);
    }
    return { success: true, violations: [] };
  }
}
```

**ポイント:**
- `scope` はオプショナルなので、既存のプラグインは変更なしで動作します。

#### 検証プラグインによる Cloud Assembly への新規ファイル作成を許可 ([#38007](https://github.com/aws/aws-cdk/pull/38007))

**この機能でできること:**
- 検証プラグインのコントラクトが「プラグインは Cloud Assembly を変更してはならない」から「プラグインは Cloud Assembly に**プラグイン実行前から存在していたファイル**を変更・削除してはならない」に緩和されました。
- これにより、SARIF レポートやカスタム出力ファイルなどの**新規ファイル**を Cloud Assembly に書き出せるようになります。
- 既存ファイルの**変更**や**削除**は引き続き検出されエラーになります。

**ポイント:**
- 内部的にディレクトリ全体のフィンガープリントから、個別ファイルごとの SHA-256 ハッシュ管理に変更されています。

#### Validation Report に抑止された違反を含める ([#38009](https://github.com/aws/aws-cdk/pull/38009))

**この機能でできること:**
- `acknowledge()` で抑止された違反 (suppressed violations) も、`validation-report.json` の `suppressedViolations` 配列にプラグイン単位で記録されるようになりました。
- 各エントリには違反の詳細に加えて、`acknowledgedId`、`reason`、`acknowledgedAt` (acknowledge() を呼んだ construct のパス) が含まれます。
- 抑止された違反は引き続きアクティブな `violations` リスト・整形済み出力・成功/失敗判定からは除外されます。

**ポイント:**
- セキュリティ・コンプライアンス監査用の監査証跡として活用できます。

### `CfnResource.addPropertyOverride` のスタックトレース記録 ([#38072](https://github.com/aws/aws-cdk/pull/38072))

**この機能でできること:**
- `CDK_DEBUG` を有効化したときに、L1 のプロパティ Setter や Box 化されたプロパティだけでなく、Escape Hatch の `CfnResource.addPropertyOverride()` 呼び出しもスタックトレースが記録されるようになりました。
- Cloud Assembly 内に書き出されるプロパティ割り当てメタデータから、Escape Hatch 経由のオーバーライドの呼び出し元も追跡できます。

**使用例:**

```bash
CDK_DEBUG=true npx cdk synth
```

```typescript
const bucket = new s3.Bucket(this, 'MyBucket');
const cfnBucket = bucket.node.defaultChild as s3.CfnBucket;
// 以下のオーバーライドの呼び出し位置がメタデータに記録される
cfnBucket.addPropertyOverride('VersioningConfiguration.Status', 'Enabled');
```

**ポイント:**
- デバッグ時に「どのコードがどのプロパティを設定したか」を追跡しやすくなります。

### 合成が遅い場合のパフォーマンスカウンタ出力 ([#38004](https://github.com/aws/aws-cdk/pull/38004))

**この機能でできること:**
- 合成 (synthesis) が遅い (現状: スタックあたり 10 秒超) と判断された場合、`$CDK_PERF_COUNTERS_FILE` 環境変数で指定されたパスにパフォーマンスカウンタを書き出すようになりました。
- CDK CLI 側でテレメトリとして送信される (またはテレメトリ無効時はスキップされる) 想定です。
- 合成内部の主要 API (アセットバンドル、ファイルシステムアクセス、サブプロセス起動など) の所要時間が記録されます。

**使用例:**

```typescript
import { App } from 'aws-cdk-lib';

// パフォーマンスレポートを完全に無効化したい場合
const app = new App({
  performanceReporting: false,
});
```

```json
// または cdk.json でコンテキスト設定
{
  "context": {
    "aws:cdk:performance-reporting": false
  }
}
```

**ポイント:**
- 出力例: `phase:Construction`、`bundle:NodejsFunction`、`DockerImage.fromBuild`、`phase:Synthesis`、`Stack.resolve` などの所要時間を記録。
- `printPerfCounters()` を直接呼んでローカルでプロファイリングすることも可能です。

### L1 CloudFormation リソース定義の更新 ([#37993](https://github.com/aws/aws-cdk/pull/37993))

**この機能でできること:**
- `@aws-cdk/aws-service-spec` の最新 CloudFormation Resource Schema に追従し、各サービスの L1 (Cfn) リソースに新しいプロパティ・型・enum 値が追加されました。
- 主な追加例:
  - **Application Signals:** `AWS::ApplicationSignals::ServiceLevelObjective` に Composite SLI (`CompositeSliConfig`、`CompositeSliComponent`、`SelectionConfig`) と外部メトリクスソース (`MetricSource`) のサポートが追加されました。
  - **AppStream:** `AWS::AppStream::StackUserAssociation` の primary identifier が `Id` ベースに変更されました。
  - **PCA Connector AD:** `AWS::PCAConnectorAD::ServicePrincipalName` および `TemplateGroupAccessControlEntry` で複数のプロパティが必須化 (詳細は破壊的変更参照)。
  - その他、各サービスの最新スキーマに基づく型定義の同期。

**ポイント:**
- L1 リソースは公開 CloudFormation Resource Schema から自動生成されます。利用しているサービスの L1 リソースを使う際は、IDE 補完で新規プロパティを確認できます。

## バグ修正

### `aws-cdk-lib`: トークン解決を約 25% 高速化 ([#37920](https://github.com/aws/aws-cdk/pull/37920))
- 特定のフィールドにトークンが含まれていないと素早く判定できる場合、低速パスをバイパスするように改善されました。
- 大規模アプリで合成時間が約 1 秒短縮されます。

### `autoscaling`: `ScheduledAction.endTime` の使用は危険 ([#38014](https://github.com/aws/aws-cdk/pull/38014))
- `ScheduledAction.endTime` プロパティを使うと CloudFormation スタックドリフトが発生し、デプロイが壊れる可能性があるため、利用が非推奨であることが明示されました。

### `bedrockagentcore`: `allowlistedHeaders` のパターンを CFN スキーマに合わせて緩和 ([#37969](https://github.com/aws/aws-cdk/pull/37969))
- `Authorization` と `X-Amzn-Bedrock-AgentCore-Runtime-Custom-*` 以外のヘッダーが指定できなかった問題を修正。CFN スキーマと同じ `^[A-Za-z][A-Za-z0-9_-]{0,255}$` を許容するようになりました。

### `cloudwatch`: メトリクス数式の検証で引用符内文字列を未知の識別子として誤検出 ([#37977](https://github.com/aws/aws-cdk/pull/37977))
- `LAMBDA("${Token[TOKEN.1937]}", "gc", "count")` のような引用符内の文字列が `usingMetrics` の未登録識別子として誤って警告されていた問題を修正しました。引用符内のワードは検証対象から除外されます。

### `core`: cross-region SSM writer がスタック更新時にパラメータをオーファン化する ([#38059](https://github.com/aws/aws-cdk/pull/38059))
- スタック更新でリソースが置換される際、cross-region SSM writer のカスタムリソースハンドラが「使用中チェック」で Delete 操作を阻止し、最終的に CloudFormation がリソースを放棄してしまい、SSM パラメータが永久にオーファン化される問題を修正しました。
- 修正後、Create / Update / Delete のいずれの操作でも検証なしに目的の操作を実行するようになりました。

### `core`: 検証で boolean 文字列 `"false"` を正しく扱うよう修正 ([#37989](https://github.com/aws/aws-cdk/pull/37989))
- CLI から `--context key=value` で渡されるコンテキスト値は文字列として届きますが、`?? true` のチェックでは文字列 `"false"` が truthy と評価されるため、`failSynthOnValidationErrors` が誤って発火していた問題を修正しました。
- `getBooleanContext` ヘルパーを導入し、boolean `false` と文字列 `"false"` の両方を正しく扱うようになりました。

### `integ-tests`: `onTimeout` で `responseURL` がログ出力される ([#37972](https://github.com/aws/aws-cdk/pull/37972))
- `@aws-cdk/integ-tests-alpha` の `onTimeout` ハンドラが、CloudFormation カスタムリソースイベントを失敗理由にシリアライズする際に、署名付き S3 URL (一時的な STS セッショントークン、アクセスキー、シグネチャを含む) も含めて出力していた問題を修正しました。
- `handler()` / `isComplete()` と同じパターンで `ResponseURL` をログ出力前に redact するようにしました。

### `lambda-nodejs`: bundling が `..` を含むエントリパスを誤って拒否 ([#38022](https://github.com/aws/aws-cdk/pull/38022))
- v2.256.x で導入されたパス検証が、ファイル名やディレクトリ名に `..` を含むパス (例: `app..js`、pnpm の `node_modules/.pnpm/file+..+other+pkg+0.0.1/...`) も誤って拒否していた回帰を修正しました。
- `String.includes('..')` ではなく、先頭が `..` セグメントの場合のみ拒否するようになりました。

### `lambda-nodejs`: パフォーマンスカウンタ e2e テストのファイル名修正 ([#38033](https://github.com/aws/aws-cdk/pull/38033))
- 内部 e2e テストで参照していたファイル名を、新しい CLI が出力するファイル名に合わせる修正。エンドユーザーへの直接的な影響はありません。

### `custom-resource-handlers`: 生成された Lambda の deterministic なアセットハッシュ ([#37634](https://github.com/aws/aws-cdk/pull/37634))

**修正内容:**
- カスタムリソースハンドラの Lambda (`aws-ses` の drop-spam、`aws-logs` の log-retention など) が `Code.fromAsset()` 経由でステージングされる際、`AssetStaging` のフィンガープリントにファイルシステムメタデータ (mtime、mode) が含まれていたため、別マシンで合成すると同じソースから別の S3 オブジェクトキーが生成され、`cdk diff` が無意味な Lambda アセット更新を表示する問題を修正しました。
- ハンドラディレクトリのコンテンツのみをハッシュした SHA-256 を `assetHash` として明示的に渡すようになり、合成結果が完全に decisive になります。

### `glue-alpha`: Ray Jobs を非推奨化 ([#38055](https://github.com/aws/aws-cdk/pull/38055))

**修正内容:**
- AWS Glue for Ray が 2026 年 4 月 30 日に新規顧客受付終了 + メンテナンスモード移行となったため、`RayJob`、`RayJobProps`、`Runtime.RAY_TWO_FOUR`、`JobType.RAY` が `@deprecated` でマークされました。
- 移行先として Amazon EKS + KubeRay Operator が推奨されています。

### `glue-alpha`: PySpark / Scala Spark ETL Job に `notifyDelayAfter` を復元 ([#37815](https://github.com/aws/aws-cdk/pull/37815))

**修正内容:**
- 以前のリファクタで `PySparkEtlJob` と `ScalaSparkEtlJob` から誤って削除されていた `notifyDelayAfter` プロパティを復元しました。Flex バリアントとパリティが取れていなかったので、それを揃えています。

```typescript
import * as glue from '@aws-cdk/aws-glue-alpha';
import * as cdk from 'aws-cdk-lib';

new glue.PySparkEtlJob(this, 'EtlJob', {
  // ...
  notifyDelayAfter: cdk.Duration.minutes(30),  // 復活したプロパティ
});
```

### `integ-tests-alpha`: アサーション失敗時の出力を簡潔化 ([#37974](https://github.com/aws/aws-cdk/pull/37974))

**修正内容:**
- アサーションが安定しない場合の失敗メッセージに、CloudFormation リクエスト全体が含まれていて読みにくい問題を修正しました。カスタムリソースに渡された入力のみが表示されるようになります。

### `mediapackagev2-alpha`: `OriginEndpoint` の `cdnAuth` が必要なポリシーを自動生成 ([#38013](https://github.com/aws/aws-cdk/pull/38013))

**修正内容:**
- `OriginEndpoint` に `cdnAuth` を設定しても、AWS のドキュメントが要求する gating policy statement (`mediapackagev2:RequestHasMatchingCdnAuthHeader` 条件を要求) が生成されておらず、エンドポイントポリシーが実質的に機能していなかった問題を修正しました。
- 修正後は `cdnAuth` を設定すると、`OriginEndpointPolicy` に gating statement と `CdnAuthConfiguration` ブロックの両方が自動的に含まれます。

**Alpha モジュールの破壊的変更:**
- `OriginEndpoint.addToResourcePolicy()` から optional な第 2 引数 `cdnAuth` が削除されました。`cdnAuth` は `OriginEndpointProps` で直接設定してください。
- `MediaPackageV2OriginProps.cdnAuth` プロパティも削除されました。

## 破壊的変更

### `aws-pcaconnectorad`: 複数 L1 リソースのプロパティが必須化 ([#37993](https://github.com/aws/aws-cdk/pull/37993))

L1 リソースは公開 CloudFormation Resource Schema から自動生成されており、CloudFormation の実態に合わせた変更です。本来必須であった項目が型定義上も必須になりました。

**変更前:**

```typescript
import * as pcaconnectorad from 'aws-cdk-lib/aws-pcaconnectorad';

// ConnectorArn / DirectoryRegistrationArn はオプショナルだった
new pcaconnectorad.CfnServicePrincipalName(this, 'Spn', {
  // ...
});

// GroupSecurityIdentifier / TemplateArn はオプショナルだった
new pcaconnectorad.CfnTemplateGroupAccessControlEntry(this, 'Acl', {
  accessRights: { /* ... */ },
});
```

**変更後:**

```typescript
import * as pcaconnectorad from 'aws-cdk-lib/aws-pcaconnectorad';

new pcaconnectorad.CfnServicePrincipalName(this, 'Spn', {
  connectorArn: 'arn:aws:pca-connector-ad:us-east-1:123456789012:connector/...',     // 必須
  directoryRegistrationArn: 'arn:aws:pca-connector-ad:us-east-1:123456789012:...',  // 必須
});

new pcaconnectorad.CfnTemplateGroupAccessControlEntry(this, 'Acl', {
  groupSecurityIdentifier: 'S-1-5-...',  // 必須
  templateArn: 'arn:aws:pca-connector-ad:us-east-1:123456789012:template/...',  // 必須
  accessRights: { /* ... */ },
});
```

**移行方法:**
- `CfnServicePrincipalName` で `ConnectorArn` および `DirectoryRegistrationArn` を必ず指定してください。
- `CfnTemplateGroupAccessControlEntry` で `GroupSecurityIdentifier` および `TemplateArn` を必ず指定してください。
- これらは元々 CloudFormation 上は必須として扱われていた項目であり、CDK の型定義が実態に合わせて修正されたものです。

## まとめ

ポリシー検証フレームワークの大幅な拡張、cross-stack 参照を細かく制御する新 API、EKS Cluster の `deletionProtection` プロパティ、ElastiCache の enum-like class への移行など、stable モジュールにも実用的な機能が多く追加された大型リリースです。`aws-pcaconnectorad` の L1 を直接利用している場合は破壊的変更への対応が必要なため、アップグレード前に確認してください。
