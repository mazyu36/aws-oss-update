## 概要

このリリースでは、L1 CloudFormation リソース定義の更新と、jsii ランタイム経由で渡される外部スタックトレースをメタデータや `ConstructError` に取り込む機能が追加されました。バグ修正としては、Docker ビルドのキャッシュ判定、デコレータパスを除外したスタックトレース、weak cross-stack 参照のリスト型属性対応などが含まれます。

**リリース:** [v2.260.0](https://github.com/aws/aws-cdk/releases/tag/v2.260.0)

---

## 新機能

### L1 CloudFormation リソース定義の更新 ([#38151](https://github.com/aws/aws-cdk/pull/38151))

**この機能でできること:**
- `@aws-cdk/aws-service-spec` の最新変更が取り込まれ、L1 (`Cfn*`) リソース定義が更新されました。
- 例: `AWS::AppStream::Stack` に `ContentRedirection` プロパティが追加、`AWS::ApiGatewayV2::Integration` に `arnTemplate` が定義、複数リソースで `primaryIdentifier` の修正、属性の追加・削除などが行われました。

**ポイント:**
- L2 コンストラクトの API には影響しません。L1 を直接利用している場合は、追加されたプロパティを利用できるようになります。
- 一部リソースで `primaryIdentifier` が変更されており、`importValue` 系のロジックを実装している場合は確認が推奨されます。

---

### `core`: `ConstructError` に外部スタックトレースを追加 ([#38131](https://github.com/aws/aws-cdk/pull/38131))

**この機能でできること:**
- jsii ランタイム経由で提供される外部スタックトレース (例: Python / Java / .NET / Go バインディングからのコールスタック) が利用可能な場合、`ConstructError` の内部トレースに付加されるようになりました。
- これにより、CDK ライブラリ内部で発生したエラーが、ユーザーアプリケーション側 (Python など) のどの行から呼び出されたものかを特定しやすくなります。

**ポイント:**
- TypeScript / JavaScript で CDK アプリを書いている場合は影響ありません (jsii ランタイムを経由しないため) 。
- jsii ホスト側のフレーム (`Kernel._Kernel_create` など) は自動的にフィルタされます。

---

### `core`: 利用可能な場合は外部スタックトレースをメタデータに追加 ([#38124](https://github.com/aws/aws-cdk/pull/38124))

**この機能でできること:**
- jsii ランタイムが [スタックトレースをワイヤー越しに送信](https://github.com/aws/jsii/pull/5153) してきた場合、それをコンストラクトの `aws:cdk:creationStack` メタデータに追加するようになりました。
- jsii ホストはグローバル状態 (`'jsii.context.hostStackTrace'` シンボル) を介してこの情報を提供します。

**最終的なメタデータの例:**

```json
{
  "type": "aws:cdk:creationStack",
  "data": [
    "...aws-cdk-lib, jsii runtime, node internals...",
    "(no user code in 10 frames, use --stack-trace-limit to capture more)",
    "__init__ (/Users/otaviom/my_app/my_app/my_app_stack.py:16:0)",
    "<module> (/Users/otaviom/my_app/app.py:10:0)"
  ]
}
```

**ポイント:**
- 合成された CloudFormation テンプレート内のメタデータに、Python など jsii バインディング側のソース位置 (ファイル名・行番号) が記録されます。
- デバッグやテンプレート解析時に、コンストラクトを生成したユーザーコードを追跡しやすくなります。

## バグ修正

### `bundling`: 既にビルド済みの場合は Docker ビルドをスキップ ([#38134](https://github.com/aws/aws-cdk/pull/38134))
- Docker イメージのバンドル時、ローカル Docker デーモンに既にイメージがキャッシュされている場合は `docker build` をスキップするようになりました (`docker image inspect <tag>` で判定) 。
- 以前 ([#37951](https://github.com/aws/aws-cdk/pull/37951)) 同様の最適化が入った後にリバート ([#38116](https://github.com/aws/aws-cdk/pull/38116)) されていましたが、タグを Docker コンテキストの「内容」ベース (ディレクトリ名やオプションだけでなく、ファイル内容を含む) で生成する修正と合わせて再導入されました。
- `--build-context` (追加のビルドコンテキスト) を指定した場合は、対象が広すぎるためキャッシュ判定はスキップされます (`LambdaNodejsFunction` などの代表的なユースケースでは利用していないため影響はありません) 。
- 効果として、`cdk synth` / `cdk deploy` を繰り返した際の Docker ビルドが大幅に高速化されます。

---

### `core`: スタックトレースからデコレータパスを除外 ([#38130](https://github.com/aws/aws-cdk/pull/38130))
- CDK ライブラリ内で収集される「クリーンアップ済み」スタックトレースに、デコレータ用のファイル (`prop-injectable.ts` など) が紛れ込む問題を修正しました。
- これらのフレームをトレースから除外し、ユーザーに表示するスタックトレースが本来のユーザーコード位置を示すようになります。

---

### `core`: weak cross-stack 参照がリスト型属性で失敗する問題 ([#37948](https://github.com/aws/aws-cdk/pull/37948))
- `Fn::GetStackOutput` を使う weak / both cross-stack 参照で、`STRING_LIST` 型の属性を扱う際に CloudFormation が「Every Value member must be a string」とエラーを返していた問題を修正しました。
- Producer 側の出力は `Fn::Join("||", ...)` でリストを文字列にシリアライズし、Consumer 側で `Fn::Split("||", Fn::GetStackOutput(...))` によりリストに復元されます。

**影響を受けていた例:**

```typescript
import * as ec2 from 'aws-cdk-lib/aws-ec2';

const vpc = new ec2.Vpc(producerStack, 'Vpc', {
  ipAddresses: ec2.IpAddresses.cidr('10.0.0.0/16'),
});
const cidrAssociation = new ec2.CfnVPCCidrBlock(producerStack, 'SecondaryCidr', {
  vpcId: vpc.vpcId,
  cidrBlock: '10.1.0.0/16',
});

// 修正前: weak 参照だと List 属性 (attrCidrBlockAssociations) でデプロイが失敗していた
// 修正後: 正しく Fn::Join / Fn::Split を介してリストが伝搬される
new SomethingThatUses(consumerStack, 'C', {
  cidrBlocks: vpc.attrCidrBlockAssociations,
});
```

**ポイント:**
- weak / both 参照を有効化していて、リスト型属性を cross-stack で参照していた構成で問題が解消されます。
- `Fn::GetStackOutput` を他の組み込み関数の引数として使う構成は、すべてのリージョンでサポートされているわけではない点に留意してください。

---

### `lambda-nodejs`: Docker バンドリングのファイル操作で `posixShellEscape` を再利用 ([#38133](https://github.com/aws/aws-cdk/pull/38133))
- Docker バンドリング経路でシェルコマンドを組み立てる際、引数のエスケープ方式が他処理と統一されていなかったファイル操作部分を、既存の `posixShellEscape` を使う形に揃えました。
- スペースや特殊文字を含むパスを扱う際の挙動が安定し、リグレッションテストも追加されています。

## まとめ

このリリースは、L1 リソース定義の更新と、Python など jsii バインディング利用時のスタックトレース表示・メタデータ拡充が中心です。バグ修正では Docker バンドリングの高速化や weak cross-stack 参照のリスト対応など、運用面での改善が含まれています。
